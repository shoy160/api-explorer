## 4. TypeScript 特定规范

### 4.1 `interface` vs `type`

| 场景 | 推荐使用 | 说明 |
|-----|---------|------|
| 定义对象结构 | `interface` | 支持声明合并 |
| 联合类型/交叉类型 | `type` | 更灵活的类型组合 |
| 简单类型别名 | `type` | 如 `type ID = string` |

### 4.2 禁止的写法

| 禁止内容 | 替代方案 | 示例 |
|---------|---------|------|
| `any` | 具体类型或 `unknown` | `const data: unknown = fetchData()` |
| 非空断言 `!` | 类型守卫或可选链 | `data?.name` |
| `as` 类型断言 | 类型守卫或正确的类型定义 | 完善类型定义 |

**错误示例 ❌**：
```typescript
const data: any = await fetch('/api');
const name = data!.name;
```

**正确示例 ✅**：
```typescript
interface ApiResponse {
  name?: string;
}
const data: ApiResponse = await fetch('/api');
const name = data.name ?? 'default';
```

### 4.3 类型导入

必须使用 `import type` 导入类型：

```typescript
import type { OpenApiSchema } from './types/openapi';
import { formatData } from './utils/format';
```

## 5. React 特定规范

### 5.1 组件设计原则

| 原则 | 说明 |
|-----|------|
| 单一职责 | 一个组件只做一件事 |
| 无副作用 | 纯展示组件不包含业务逻辑 |
| 可复用性 | 通用组件应易于复用 |
| Props 类型化 | 必须定义 Props 接口 |

### 5.2 性能优化原则（CRITICAL）

#### 5.2.1 消除请求瀑布

| 规则 | 说明 |
|-----|------|
| `async-parallel` | 使用 `Promise.all()` 并行执行独立操作 |
| `async-defer-await` | 将 await 移到实际使用的分支中 |
| `async-cheap-condition` | 在 await 之前检查廉价的同步条件 |

**错误示例 ❌**：
```typescript
const user = await fetchUser();
const posts = await fetchPosts(user.id);
```

**正确示例 ✅**：
```typescript
const [user, posts] = await Promise.all([
  fetchUser(),
  fetchPosts()
]);
```

#### 5.2.2 Bundle 大小优化

| 规则 | 说明 |
|-----|------|
| `bundle-barrel-imports` | 直接导入，避免 barrel 文件 |
| `bundle-dynamic-imports` | 对重量级组件使用动态导入 |
| `bundle-defer-third-party` | 在 hydration 后加载分析/日志库 |

**正确示例 ✅**：
```typescript
import { Button } from 'antd/button';

const HeavyChart = React.lazy(() => import('./HeavyChart'));

<Suspense fallback={<Loading />}>
  <HeavyChart />
</Suspense>
```

#### 5.2.3 渲染优化

| 规则 | 说明 |
|-----|------|
| `rerender-memo` | 将昂贵的计算提取到 memoized 组件 |
| `rerender-no-inline-components` | 不要在组件内部定义组件 |
| `rerender-transitions` | 使用 `startTransition` 处理非紧急更新 |
| `rerender-use-deferred-value` | 使用 `useDeferredValue` 延迟昂贵渲染 |

**正确示例 ✅**：
```typescript
const expensiveResult = useMemo(() => {
  return computeExpensiveValue(data);
}, [data]);

const handleClick = useCallback(() => {
  setCount(prev => prev + 1);
}, []);

const [searchTerm, setSearchTerm] = useState('');
const [isPending, startTransition] = useTransition();

const handleSearch = (value: string) => {
  startTransition(() => {
    setSearchTerm(value);
  });
};
```

#### 5.2.4 服务端性能

| 规则 | 说明 |
|-----|------|
| `server-cache-react` | 使用 `React.cache()` 进行请求去重 |
| `server-serialization` | 最小化传递给客户端组件的数据 |

```typescript

interface ComponentProps {
  title: string;
  onClick?: () => void;
}

  const [count, setCount] = useState(0);
```

### 5.4 Hooks 使用规范

- 只在组件顶层调用 Hooks
- 不要在循环、条件或嵌套函数中调用
- 自定义 Hook 必须以 `use` 开头
- 使用 `useMemo`/`useCallback` 避免不必要的重渲染
- 使用 `useRef` 存储瞬态值（如滚动位置、定时器 ID）
- 使用 `useState` 的函数形式处理昂贵的初始值

**正确示例 ✅**：
```typescript
const [data, setData] = useState(() => expensiveInit());

const timerRef = useRef<number | null>(null);

useEffect(() => {
  timerRef.current = window.setInterval(() => {
  }, 1000);
  
  return () => {
    if (timerRef.current) {
      clearInterval(timerRef.current);
    }
  };
}, []);
```

## 8. Git 工作流

### 8.1 分支策略

采用 **Git Flow** 变体，适合小团队协作：

| 分支类型 | 命名格式 | 说明 |
|---------|---------|------|
| 主分支 | `main` | 生产环境代码 |
| 开发分支 | `develop` | 开发集成分支 |
| 功能分支 | `feature/<name>` | 新功能开发 |
| Bug 修复 | `fix/<name>` | Bug 修复 |
| 热修复 | `hotfix/<name>` | 生产环境紧急修复 |
| 发布分支 | `release/<version>` | 版本发布准备 |

### 8.2 分支命名规范

```
<type>/<description>
```

**type 列表**：
- `feature`：新功能
- `fix`：Bug 修复
- `refactor`：代码重构
- `style`：样式调整
- `docs`：文档更新
- `test`：测试相关
- `chore`：杂务（构建、工具等）
- `perf`：性能优化
- `ci`：CI/CD 配置

### 8.3 Conventional Commits 规范

**格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 列表**：
| type | 说明 |
|-----|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 代码重构（不改变功能） |
| `style` | 样式调整（不影响代码逻辑） |
| `docs` | 文档更新 |
| `test` | 添加/修改测试 |
| `chore` | 构建/工具/依赖更新 |
| `perf` | 性能优化 |
| `ci` | CI 配置变更 |

**scope 列表**：
- `api`：API 解析相关
- `ui`：界面组件相关
- `store`：状态管理相关
- `service`：服务层相关
- `utils`：工具函数相关

**提交示例**：
```
feat(api): 添加 OpenAPI 3.0 规范支持

- 集成 swagger-parser 库
- 支持解析 components 中的 schema
- 处理 $ref 引用解析

Closes #123
```

```

- 调整媒体查询断点
- 修复菜单折叠状态切换逻辑
```

## 9. 代码注释规范

### 9.1 注释原则

| 场景 | 是否需要注释 | 说明 |
|-----|-------------|------|
| 复杂业务逻辑 | ✅ | 解释"为什么"而不是"做了什么" |
| 算法实现 | ✅ | 说明算法思路和复杂度 |
| 临时代码 | ✅ | 使用 TODO/FIXME 标记 |
| 自文档化代码 | ❌ | 如 `const userName = 'John'` |
| Getter/Setter | ❌ | 简单的访问器不需要注释 |

### 9.2 注释风格

```typescript
/**
 * 解析 OpenAPI 文档
 * @param url - 文档 URL 或本地路径
 * @returns 解析后的文档对象
 * @throws {Error} 当文档格式无效时
 */
export async function parseOpenApi(url: string): Promise<OpenApiDocument> {
}
```

### 9.3 特殊标记

| 标记 | 含义 | 示例 |
|-----|------|------|
| `TODO` | 需要完成的任务 | `// TODO: 后续支持 XML 格式` |
| `FIXME` | 需要修复的问题 | `// FIXME: 处理大文件内存溢出` |
| `HACK` | 临时解决方案 | `// HACK: 临时兼容旧版本 API` |

## 10. 错误处理规范

### 10.1 服务端错误处理

```typescript
try {
  const response = await apiClient.get('/api/data');
  return response.data;
} catch (error) {
  if (axios.isAxiosError(error)) {
    const status = error.response?.status;
    if (status === 401) {
      navigate('/login');
    } else if (status === 404) {
      console.warn('Resource not found');
      return null;
    } else {
      message.error('请求失败，请稍后重试');
    }
  } else {
    console.error('Unexpected error:', error);
    message.error('系统错误');
  }
}
```

### 10.2 客户端错误处理

- 使用类型守卫进行类型检查
- 避免直接暴露错误堆栈给用户
- 提供友好的错误提示

```
│  components │    pages    │    store    │   services  │

│             │             │     │                      │
│             └─────────────┴─────┴───────────┐          │
│                                            ↓          │
│                                   ┌─────────┴──┐       │
│                                   │   utils    │       │
│                                   │   types    │       │
│                                   │   data     │       │
│                                   └────────────┘       │
└─────────────────────────────────────────────────────────┘
```

**合法依赖方向**：
- components → pages, store, hooks, utils, types
- pages → components, store, hooks, utils, types
- store → services, utils, types
- services → utils, types
- hooks → store, services, utils, types
- utils → types

**禁止的依赖方向**：
- utils → components / pages / store / services / hooks
- types → 任何其他模块
- components → 其他 components（除非是通用组件）

## 16. 安全编码规范

| 安全要点 | 说明 |
|-----|------|
| 输入校验 | 对所有用户输入进行校验和 sanitize |
| SQL 注入 | 使用 ORM 参数化查询（本项目无后端，前端需注意 URL 参数） |
| XSS 防护 | 使用 React 自动转义，避免 `dangerouslySetInnerHTML` |
| 密钥管理 | 不在前端代码中存储敏感信息 |
| 文件上传 | 本项目暂不支持文件上传 |
| 认证检查 | 使用拦截器统一处理 Token 失效 |
| HTTPS | 生产环境必须使用 HTTPS |
